# 側展間分支畫廊 — 設計文件

日期：2026-07-16
狀態：已與使用者確認方向，待撰寫實作計畫

## 背景與目標

目前 `app-gallery.js` 的 3D 畫廊是一條沿 Z 軸延伸的單一直線走廊（`buildShell`），作品以「兩兩一對、左右交替」的方式掛在走廊兩側牆上（`slotTransform` / `buildFrames`），走廊長度 `hallLen` 會隨作品數量自動變長（`rebuildScene` 內 `needed` 的計算）。

使用者希望：作品量變多時，空間不要只是一直線變長，而是要有「轉彎、像真的畫廊」的感覺。經討論後，選定「主走廊 + 側展間」的方案：作品累積到一定數量後，額外的作品會被分流到主走廊沿途開出的側展間裡，而不是讓主走廊無限延伸。

## 核心邏輯

### 1. 分區規則

- 新增常數 `ROOM_CAPACITY = 6`：每 6 件作品為一個「展區」（沿用現有 `slotTransform` 的兩兩配對邏輯，一個展區 = 3 對）。
- 第 1 個展區（前 6 件作品）行為與現在完全一致：直接掛在主走廊兩側牆上，主走廊長度固定為可容納這 6 件作品的長度（不再隨作品數逐件變長）。
- 從第 2 個展區開始，每滿 `ROOM_CAPACITY` 件作品，就在主走廊沿途開一間「側展間」，把整批作品都放進該側展間內展示，而不掛在主走廊牆上。
- 側展間沿主走廊延伸方向依序排列，開口左右交替（第 1 間開左邊、第 2 間開右邊……），透過一個牆面開口 + 短連接通道從主走廊接進房間。
- 主走廊長度只隨「側展間數量」小幅增加（每間側展間需要沿走廊佔一段開門的牆面 + 前後留白），不再隨作品總數線性增長。
- 若最後一個展區作品數不足 6 件（例如總數是 14 件 → 第一區 6 件在主走廊、第二區 6 件在側展間 1、剩下 2 件在側展間 2），側展間仍會開出來，只是房間內的作品較少、房間尺寸可依實際件數微調（但不追求動態花俏，維持固定房間外殼尺寸即可，內部作品靠中間集中排列）。

### 2. 空間資料結構

新增一個「區域清單」資料結構，取代目前單一走廊矩形的概念：

```js
{
  id: "main" | "room-1" | "room-2" | ...,
  kind: "corridor" | "room" | "connector",
  bounds: { minX, maxX, minZ, maxZ },  // 世界座標下的可行走矩形範圍
  connectsTo: ["main", ...]            // 相鄰、可跨越的區域 id
}
```

- 主走廊、每間側展間、每條連接通道，各自對應一筆區域資料。
- 側展間與其連接通道用該側展間在主走廊上的分支點座標推導出各自的世界座標範圍（房間往左側分支就往 -X 方向延伸，右側則往 +X 方向）。
- 這份區域清單在每次 `rebuildScene` 重新產生（隨作品數變化），供建構幾何體與攝影機碰撞共用同一份資料，避免兩邊算出不一致的座標。

### 3. 幾何建構

- `buildShell` 需要擴充：主走廊殼體維持現有做法（牆、地板、天花板、燈光），但長度改用新規則計算；此外要在對應側展間分支點的走廊牆上挖出開口（該段不生成牆面 mesh），並新建每間側展間自己的殼體（四面牆、地板、天花板、房間內燈光，尺寸比主走廊小一號），以及連接通道的殼體。
- `buildFrames` 的掛畫邏輯（`slotTransform` 目前是純線性 Z 座標計算）需要拆成「依區域計算」版本：主走廊展區用現有算法；側展間展區改用房間局部座標系（房間中心為原點、兩側牆），再轉換成世界座標。掛畫視覺本身（相框、圖片、標示牌）不需要改動，只是換算座標的方式要能對應到房間牆面而非走廊牆面。
- 中央展示台（`buildFeatured` / `plinthZ`）維持在主走廊內，不受側展間影響，邏輯不變。

### 4. 攝影機移動與碰撞

現行做法（`update(dt, t)` 內）是把攝影機 X/Z 各自 clamp 在單一矩形範圍，另外對中央展示台做一個簡單的「實心箱推開」處理。

改為「區域碰撞」：

- 每一影格，先用目前攝影機座標找出玩家「目前所在區域」（沿用上一影格判定結果作為預設值，找不到才重新掃描整份區域清單，避免邊界抖動）。
- 計算移動後的候選新座標；若新座標落在目前所在區域的 `bounds` 內，直接採用。
- 若新座標超出目前區域，檢查是否落在該區域 `connectsTo` 中任一相鄰區域的 `bounds` 內（門口/ 連接通道會讓相鄰兩個區域的矩形有重疊帶，因此可以連續判斷），若是則允許移動並把「目前所在區域」更新成該相鄰區域；否則沿用現有 `clamp` 邏輯把候選座標夾回目前區域邊界。
- 中央展示台的實心箱推開邏輯不變，仍套用在主走廊區域內。
- 進場動畫（`introActive` 分支）與「雙擊聚焦到某作品」的鏡頭 tween（`camZTween`）不受影響，因為它們的目標點必定落在合法區域內（主走廊或對應側展間），不需要額外處理。

### 5. 小地圖

`drawMinimap` 目前把整個場地當成一個矩形（`hallLen` × `HALL_W`）等比縮放繪製。改為：

- 依區域清單，畫出主走廊的長條矩形，以及每間側展間從走廊分支出去的小矩形（含連接通道），呈現樹狀平面圖。
- 縮放比例改用「所有區域座標的整體外接框」而非單純 `hallLen` × `HALL_W`。
- 作品位置點、中央展示台位置點、玩家位置點的畫法邏輯不變，只是座標轉換函式 `mx()` / `my()` 要能吃到區域清單算出的整體外接框，而不是走廊本身的長寬。

### 6. 邊界情況

- 作品數 ≤ `ROOM_CAPACITY`（6）：只有主走廊、沒有側展間，畫面與現行版本完全一致，不應有任何行為差異（含小地圖畫法退化回單一矩形）。
- 作品數為 0：維持現有「敬請期待」空狀態邏輯，不受影響。
- 新增/刪除作品導致展區數量變動時，`rebuildScene` 會整個重建 shell + 區域清單（沿用現有「尺寸不同就整個重建」判斷方式，只是判斷條件從單一 `hallLen` 改成比較整份區域清單是否有變化）。

## 不在本次範圍內

- 不做 L 型/之字型主走廊轉彎、不做回字型迴圈（依先前確認，僅做「主走廊 + 側展間」）。
- 不依作品分類（`SECTIONS`）分房間，房間純粹依上傳時間累積批次分配。
- 側展間內部不做進一步的子分支（不會有「房間裡的房間」）。
- 不新增碰撞用的射線/物理引擎，維持既有的矩形 clamp 風格寫法。

## 測試方式

由於此專案沒有自動化測試，驗證方式為手動操作：

1. 用不同作品數量（0 件、6 件以內、7～12 件、20 件以上）跑一次 `rebuildScene`，確認主走廊與側展間數量符合預期。
2. 實際在瀏覽器中用 WASD / 觸控搖桿在主走廊與側展間之間走動，確認：
   - 無法穿牆。
   - 可以順利從主走廊走進側展間、也能走回主走廊。
   - 中央展示台附近的推開效果仍正常。
3. 檢查小地圖是否正確畫出分支形狀，玩家位置點是否隨移動正確更新。
4. 檢查雙擊聚焦作品的鏡頭 tween，在側展間內的作品也能正確運作。
